# ADR-0010 — Backend logging convention

## Status

**Accepted.** Source page `438735165`.

## Context

Every Project H backend service needs a uniform logging shape so logs are searchable, correlatable across requests, and free of PHI. Without an enforced convention, individual services drift in level usage, prefix structure, and what they capture for external calls — making operational triage and HIPAA audits both harder.

## Decision

Five logging modules, each at the level shown, with a mandatory prefix on every entry: `request_id` (UUID generated by mobile or backend if absent, carried in `X_REQUEST_ID` header), `user_id` (extracted from access token on authenticated requests), and `function` + `file` of the call site.

1. **Endpoint-execution log** (`INFO`) — start/end of resource-modifying methods, with identifiers and non-personal metadata. GET endpoints and health checks are not logged. Each endpoint opts in via a `logable` marker.
2. **HTTP request/response log** (`DEBUG`) — full JSON of request and response, URL, query parameters; only on `logable` endpoints.
3. **DB query log** (`DEBUG`) — SQL text with parameter values if safe, placeholders otherwise; auto-emitted by the ORM layer.
4. **External-system log** (`INFO`) — request and response payload (text), URL, parameters (passwords redacted) for every call to EPIC, MyChart, Cognito, SES, etc.
5. **Error log** (`ERROR`) — input parameters of the failing method, class and method name, full stack trace on a single line for CloudWatch search.

Environment variables control the level; `DEV` / `STAGE` enable `DEBUG`, `PROD` defaults to `INFO`.

## Alternatives considered

- **Single log stream with no module separation.** Pros: simpler config. Cons: forces operators to filter by substring; DB query volume drowns out endpoint signals. *Rejected* — module separation is the search affordance.
- **Trace-system-of-record only (e.g. OpenTelemetry traces, no structured logs).** Pros: native correlation. Cons: extra infra not yet provisioned; PHI-redaction policy on traces is no simpler. *Rejected for MVP*; lift if observability budget expands.

## Consequences

### Positive

- One prefix shape across services means a single CloudWatch query finds every log line for a given `request_id` end-to-end.
- The `logable` endpoint marker prevents the most expensive logs (full request/response JSON) from being written for high-volume GET / healthcheck paths.
- External-system logs capture the wire-level interaction with EPIC and Cognito, which is the most-asked-about surface during operational triage.

### Negative

- Two-level config (per-environment + per-endpoint marker) is a small ergonomics tax — devs must remember to mark new endpoints.
- DEBUG-level DB query logging in DEV adds non-trivial volume; not a problem in DEV but must stay off in PROD.

### Open

- **PHI redaction in external-system logs.** Logging full EPIC FHIR responses risks PHI ending up in CloudWatch. *Owner:* Compliance Engineer + Tech Lead. *Outcome:* policy decision before EPIC integration ships to STAGE.
